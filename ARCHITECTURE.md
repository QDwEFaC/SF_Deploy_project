# Монолит или микросервисы

Так как проект учебный для основной части (Flask + uWSGI) выбран монолит.
- для реализации обучения модели и выполнения predict не требуется независимое масштабирование компонентов
- монолит проще собрать в Docker
- микросервисы имели бы смысл при командной разработке по нескольким направлениям 

# Брокер сообщений (концепт)

При росте нагрузки на сервис можно было бы использовать RabbitMQ. Можно было бы реализовать асинхронные прогнозы - пакеты заявок складываем в очередь, воркеры читают и вызывают модель. Также можно сегметировать логику работы ядра сервера (работа с признаками, обработка предсказаний, общая маршрутизация и т.д.)

# Логирование и мониторинг (концепт)

После каждого запроса в stdout выводится строка (event, method, path, status, latency_ms) для POST /predict дополнительно (prediction, probability_default). В production такие строки собирают агенты (Filebeat, Fluent Bit), пересылают в Elasticsearch и визуализируют в Kibana (ELK-стек) или используют OpenSearch + дашборды для SLA, доли ошибок и задержек.
